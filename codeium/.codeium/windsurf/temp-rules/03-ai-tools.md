# 03 · AI 工具使用準則

> 適用於 Windsurf (Cascade)、Cursor、GitHub Copilot、Claude 等 AI 輔助開發工具。
>
> 參考來源：Anthropic Model Spec、Cursor Rules Community、Windsurf Documentation、OpenAI 提示工程指南

---

## 核心原則

**AI 是架構夥伴，不是程式碼生成機器。**

```
❌ 錯誤用法：「幫我寫一個完整的電商系統」
✅ 正確用法：「根據 spec.md 中的 User Story US-001，幫我實作 UserService.createUser() 方法，
              遵循 SOLID 原則，使用 Repository Pattern，並先撰寫測試」
```

## 提示（Prompt）工程準則

### 有效提示的結構

```
[上下文] 我正在開發 {專案名稱}，使用 {技術棧}
[任務]   需要實作 {具體功能}
[限制]   必須遵循 {規則/模式}，不使用 {排除項目}
[輸出]   請提供 {期望的輸出格式}
```

**範例**:
```
上下文：Full-stack 應用，後端使用 Laravel 11 + PostgreSQL
任務：實作 OrderRepository 的 findPendingOrders() 方法
限制：使用 Eloquent ORM，遵循 Repository Pattern，先寫測試
輸出：1) 測試程式碼 2) 實作程式碼 3) 簡短說明設計決策
```

### 提示品質規則

| 規則 | 說明 |
|------|------|
| **具體勝於模糊** | 指定函式名稱、回傳型別、邊界條件 |
| **不重複指示** | 已在規則文件中的準則無需在提示中重複 |
| **批次處理** | 將相關小變更合併為一次提示，減少來回次數 |
| **複用上下文** | 在同一對話中延續，而非開新對話重新解釋 |
| **逐步確認** | 複雜功能分步驟確認，不一次生成大量程式碼 |

## AI 輔助的使用場景分類

### 高價值場景（積極使用）

- **規格撰寫**: 協助將模糊需求轉化為結構化規格
- **初始骨架**: 建立符合架構的程式碼骨架與介面定義
- **測試生成**: 根據介面或規格生成測試用例
- **程式碼解釋**: 解釋不熟悉的程式碼邏輯
- **文件撰寫**: 生成 JSDoc / docstring
- **重構建議**: 識別違反 SOLID 的程式碼並提出改善建議
- **SQL 查詢優化**: 分析慢查詢並提供改善方案

### 謹慎使用場景（需人工審查）

- **複雜業務邏輯**: AI 可能誤解業務規則，必須仔細驗證
- **安全性相關程式碼**: 認證、授權、加密必須人工複查
- **資料庫 Migration**: 不可逆操作，必須人工確認
- **效能關鍵路徑**: AI 建議需有 benchmark 驗證

### 避免使用場景

- **整個功能的端對端生成**: 缺乏上下文，易產生架構問題
- **在未閱讀的程式碼上直接修改**: 必須先理解再修改
- **AI 驅動的除錯迴圈**: 如果連續三次 AI 未解決問題，停下來手動分析

## Windsurf (Cascade) 特定準則

### Rules 文件管理

Windsurf 會讀取 `.windsurf/rules/` 目錄中的規則文件。維護準則：

```
.windsurf/rules/
├── global.md          ← 適用所有場景的全域規則
├── specify-rules.md   ← SDD 工作流程特定規則
└── {feature}.md       ← 特定功能開發期間的臨時規則
```

**規則文件最佳實踐**:
- 每個規則文件保持單一職責（≤ 100 行）
- 使用清晰的標題和分組
- 定期清理不再適用的規則
- 使用 `@` 引用其他文件提供上下文

### Memory 管理

- **記憶體（Memory）**: 適合儲存專案架構決策、技術選型理由
- **上下文視窗**: 優先提供當前任務最相關的文件
- 避免在每次對話中重新解釋已記憶的架構決策

### 常用工作流程指令

```
/speckit.specify    ← 建立功能規格
/speckit.plan       ← 建立實作計畫
/speckit.tasks      ← 生成任務清單
/speckit.implement  ← 執行實作
/speckit.analyze    ← 一致性分析
```

## Cursor 特定準則

### `.cursorrules` 設定建議

```markdown
# 專案規則

## 核心準則
- 遵循 SOLID 原則
- 測試先行（TDD）
- 使用 TypeScript strict mode

## 程式碼風格
- 縮排：2 spaces
- 使用 async/await，避免 callback
- 匯入順序：內建 → 第三方 → 本地

## 禁止行為
- 不使用 any 型別
- 不使用 var
- 不提交未使用的匯入
```

### Cursor Composer 使用技巧

- 使用 `@file` 引用特定檔案提供上下文
- 使用 `@codebase` 讓 Cursor 搜尋相關程式碼
- 複雜任務先用 Composer 規劃，再切換到 Chat 執行

## 程式碼審查 AI 輔助

### AI 審查 ≠ 替代人工審查

AI 可以幫助：
- 識別常見的程式碼問題（未處理的 Promise、潛在 NPE）
- 檢查命名一致性
- 提醒可能遺漏的邊界條件

AI 無法替代人工審查：
- 業務邏輯正確性
- 架構一致性
- 安全漏洞的完整評估
- 效能影響的系統性評估

## AI 生成程式碼的驗收標準

在接受 AI 生成的程式碼前，必須確認：

- [ ] 理解每一行程式碼的作用（不接受「AI 寫的，應該沒問題」）
- [ ] 測試覆蓋生成的程式碼
- [ ] 無硬編碼的密鑰、密碼、URL
- [ ] 錯誤處理完整
- [ ] 符合現有程式碼風格
- [ ] 依賴的套件版本合理且無已知漏洞

## 成本最佳化

### 避免不必要的 AI 呼叫

```
✅ 適合用 AI：複雜邏輯設計、架構建議、文件生成
❌ 浪費資源：簡單的字串操作、基本的陣列處理、已有現成工具可處理的格式化
```

### 批次處理

```typescript
// ❌ 多次獨立呼叫（浪費 context）
// 第一次對話：「幫我寫 createUser 函式」
// 第二次對話：「幫我寫 updateUser 函式」

// ✅ 批次處理（效率最高）
// 「根據以下介面定義，幫我實作 UserRepository 的所有 CRUD 方法」
```

## 責任邊界

| 責任 | AI 輔助 | 人工負責 |
|------|---------|---------|
| 規格撰寫 | 協助結構化 | 業務正確性 |
| 架構設計 | 提供選項 | 最終決策 |
| 程式碼生成 | 提供草稿 | 審查與確認 |
| 測試策略 | 建議用例 | 完整性保證 |
| 安全性 | 初步掃描 | 深度審查 |
| 部署決策 | 分析影響 | 最終批准 |
